Seminario Final IS

Introduccion

Museo es un problema que explotará nuestras habilidades como futuros profesionales en nuestra rama, incluyendo como herramienta genérica el conjunto de conceptos necesarios en la organización de un proyecto, que hemos visto durante el curso, con uno de los paradigmas de la programación ágil en la cabecera. Además nos aportará y nos vinculará aún más en la práctica sobre el diseño de una base de datos aplicada pensando en las reglas del negocio que nos ofrece el cliente, en este caso un museo.

🔻 Alcance del producto

Como el museo desea presentar un catálogo con las obras de arte que se encuentran expuestas en sus salas, el producto tiene un alcance internacional, ya que mediante la página web
personas de cualquier parte del mundo tendrán acceso a la información sobre la exposición del museo para así decidir si hacen de este un destino para sus viajes vacacionales. También va a ser un producto altamente utilizado por el propio personal del museo para mantener el control del préstamo, restauración y valoración económica de las obras.

🔻 Resumen

Según los requerimientos del museo existen funciones que debe cumplir el futuro Software, como el almacenamiento de los datos, la gestión y separación del personal que accederá a la página según los roles que se encapsulan en el concepto de usuario, al igual que el procesamiento de los préstamos de obras. Todo el desarrollo de esta lógica la dividiremos en tres etapas, o dos Updates a partir de la primera entrega funcional, utilizando una metodología ágil, Srum. Los objetivos del proyecto, se lograrán siguiendo una arquitectura 3-capas, dadas las funcionalidades que se quieren lograr por separado, entre otras características del proyecto que hacen que nos favorezca dicha arquitectura para el desarrollo del software. Se pretende además que sea una aplicación multiplataforma, y qué mejor para esto que una página web para el frontend. Ofrecemos un diseño amigable para el entendimiento de la estructura de la base de datos con un Modelo Entidad Relacional Extendido (MERX), que constituye el kernel de nuestro trabajo. Básicamente la otra parte del proyecto es la aplicación de operaciones CRUD sobre estos datos cumpliendo las reglas del negocio.

Requerimientos específicos

🔻 Requerimientos Funcionales

  • Una base de datos dinámica con los siguientes datos:

    • Catálogo de obras de arte e información relacionada con cada una, como su autor y otros datos específicos.
    • Información de restauración de las obras.
    • Información de préstamos de las obras, incluye la lista de museos con los que se puede colaborar, el tiempo de préstamo y el importe pagado al museo por cada obra cedida.
    • Qué Obras se exponen en cada sala del museo.
    • Valoración de cada obra.
    • Información de autenticación

  • Un sistema de consultas a dicha base de datos que permita recibir en formato de texto los datos almacenados, incluyendo:

    • [Restaurador Jefe] Una lista de las obras que les toca ser restauradas(dada la periodicidad requerida para la restauración), así como las que ya han pasado por este proceso.
    • [Director del museo] La valoración de las obras (la suma de todas las calificaciones)

  • La existencia de una "cola de pedidos" por cada una de las obras que permita ceder una obra pedida por un museo, luego que concluya el periodo de préstamo por el cual se encontraba dicha obra en otro museo, en caso de que así ocurra.

  • El sistema debe funcionar estando incluso sin información inicial, o sea, la base de datos puede estar vacía y debe funcionar así.

🔻 Requerimientos No Funcionales

  • La aplicación no tendrá gran cantidad de niveles de profundidad en la parte visual.
  • La interfaz será minimalista, este no es objetivo del proyecto.
  • El encargado del catálogo (y solo él) será quien tenga disponible la funcionalidad de modificar o introducir datos sobre las obras.
  • El restaurador jefe (y solo el) será quien indique en el programa que una obra está en reparacion o deja de estarlo, e introduzca la información relacionada con esto.
  • Un sistema de autenticación para todos los usuarios, que otorge funcionalidades en la interfaz, dado el tipo de usuario logeado. Esto se verá más adelante en el diagrama de casos de uso.

🔻 Requerimientos de Entorno

  • No será requerimiento el sistema operativo en el que se ejecute nuestro proyecto, dado que utilizaremos desarrollo web con python, y todo esto es multiplataforma. Además nuestro cliente cuenta con los servidores sobre los que vamos a montar el Software.
  • Al ser una aplicación web el usuario podrá aceder al software desde cualquier dispositivo con navegador.

Enfoque Metodológico

Teniendo en cuenta las características de nuestro equipo, nos hace falta una metodología ágil, específicamente Scrum, ya que tenemos un Scrum Master en el equipo y como el museo esta interesado en que el proyecto se culmine rápido, su personal estará disponible para realizar reuniones sistemáticas con nuestro equipo. La frecuencia de reuniones entre los miembros del equipo será diaria y con el cliente será semanalmente, siguiendo el patron usual en esta metodología ágil.

Planificación y Riesgos

🔻 Diagrama o Tablero

Para organizar el tiempo que le dedicamos a cada actividad desde el análisis del problema hasta la instalación en el museo del sistema de gestión, nos trazamos un plan de actividades con la ayuda de un Diagrama Gant.

🔻 Hitos

Como para la realización del proyecto contamos con un plazo de 35 días, establecimos un plan de hitos con los cuales poder medir el avance del proyecto y presentarle a nuestro cliente como va el proyecto para el cual nos contrata.

Hito 1:

  • Gestionar Usuarios.
    • Agregar Usuarios
    • Editar permisos de Usuarios
    • Permitir la autenticación de usuarios con nombre y contraseña.
  • Gestionar Catálogo.
    • Actualización (eliminar, insertar obras) del catálogo por usuarios autorizados (encargado del catálogo).
    • Permitir la visualización del catálogo para todos los usuarios.

Hito 2:

  • Gestionar restauración de obras.
    • Permitir a usuarios con permiso (jefe de restauración) enviar a restauración obras de arte.
    • Permitir al jefe de restauración obtener la información correspondiente a la restauración de las obras de arte en orden cronológico inverso.

Hito 3:

  • Gestionar préstamo de obras.
    • Posibilitar el préstamo de obras de arte a otros museos por usuarios autorizados (director del museo).
    • Permitirle al director del museo el acceso a la suma total del valor de las obras de arte que tiene el museo.

Nota: Paralelamente a cada uno de los hitos trazados se irán mostrando versiones de la apariencia de la página web al cliente para que este nos haga saber sus preferencias y los cambios que le desea hacer.

🔻 Riesgos

  • Un problema común en el desarrollo de cualquier proyecto, es la pérdida de información, debido a la confusión entre versiones y problemas de almacenamiento. Para evitar esta problemática, usaremos el sistema de control de versiones GIT, y el sistema de almacenamiento remoto de GitHub.
  • Otro inconveniente podría ser la falta de comunicación entre los miembros de equipo, también entre este y el cliente.
  • Podrá afectarnos el corto tiempo requerido por la empresa para la entrega del proyecto.

Arquitectura

Dado que por lo general se requiere facilidad de testeo unitario, o sea, una separación entre capas bien definidas para una mayor limpieza en las pruebas y el mantenimiento del Software, utilizaremos una arquitectura de Dominio. Este proyecto lo definimos en las siguientes 4 capas:

  • Interfaces: Una capa de interfaz sencilla que permita una autenticación, y dada esa autenticación, el mostrado de diferentes interfaces, que coincidan con el rol del usuario, es donde residen las funcionalidades de la página web.
  • Aplicación/Casos de Uso: Atendiendo a los diferentes tipos de usuario que pueden acceder a la aplicación se conocen los distintos casos de uso, en esta capa contiene las funcionalidades de dirigir el flujo de los casos de uso.
  • Dominio: Es donde reside toda la lógica del negocio.
  • Datos: Es donde se almacenan los modelos de datos, por ejemplo los models de Django.

Patrones de Visualización y de Datos

Django fue diseñado para promover el acoplamiento débil y la estricta separación entre las piezas de una aplicación. En la capa de vista usaremos un sistema de plantillas que nos ofrece Django para poder separar la lógica de negocios de la lógica de presentación; las URLconfs de Django son un claro ejemplo de este principio en la práctica. En una aplicación Web de Django, la definición de la URL y la función de vista que se llamará están débilmente acopladas; de esta manera, la decisión de cuál debe ser la URL para una función, y la implementación de la función misma, residen en dos lugares separados. Esto permite el desarrollo de una pieza sin afectar a la otra.

Con respecto a la capa de la base de datos, aplicamos esa misma filosofía para el acceso lógico a los datos usando los modelos que ofrece Django para ejecutar código SQL y retornar estructuras de datos convenientes en Python sin la necesidad de escribir directamente en SQL. Cada modelo generalmente corresponde a una tabla única de la base de datos, y cada atributo de un modelo generalmente corresponde a una columna en esa tabla. El nombre de atributo corresponde al nombre de columna, y el tipo de campo corresponde al tipo de columna de la base de datos.

Principio SOLID

  • Como nuestro proyecto será hecho en Django toda clase que creemos sera como heredera de las ya existentes en este paquete y, respetando la separación clara de Django de models/views/templates, podremos crear con bastante facilidad clases que solo tengan una responsabilidad única en el proyecto (Principio de Responsabilidad Única o Single Responsibility Principle) .
  • Aprovechando el polimorfismo característico de Python podemos resolver la extensibilidad de nuestro proyecto. Por ejemplo, creando clases genéricas como “Usuario” y que de ella se deleguen responsabilidades como “Director del museo” o “Encargado del catálogo” se podrían crear sin demasiado problema nuevos tipos de usuarios con responsabilidades únicas y distintas de los ya existentes sin necesidad de cambiar el código ya escrito con anterioridad (Principio Open/Closed).
  • Si nos aseguramos en todo momento de usar las clases más genéricas y solo especificar en algún funcionamiento específico, se podría resolver sin problema el hecho de utilizar clases extendidas de las clases ya existentes, por lo que no habría ningún problema a la hora de incorporarlas al proyecto (Principio de Sustitución de Liskov)
  • Para asegurarnos de que no implementamos cosas que no vamos a utilizar trataremos de desacoplar funcionalidades y, por medio de la herencia múltiple de Python, utilizarlas cuando sea conveniente hacerlo evitando así que las clases implementen cosas innecesarias (Principio de segregación de interfaces)
  • Como vamos a utilizar la arquitectura de capas de Django (models/views/templates) implementaremos cada uno de estos en ficheros distintos tratando de desacoplar lo mayor posible y evitar las dependencias innecesarias de estas capas de abstracción, de esta forma evitaremos las dependencias innecesarias que podrían en un futuro sernos un problema a la hora de testear estas capas (Principio de inversión de dependencias).

Esqueleto (core) de la Aplicación

Como el softwares utilizará Django con una arquitectura dominio, el proyecto estará estructurado atendiendo a 4 capas fundamentales: los datos, el dominio, los casos de uso y las interfaces, para cada uno se crea una carpeta en la que se guardarán todods los archivos correspondientes a la capa en cuestión. En cada capa, para incrementar el desacoplamiento se separan en subcarpetas que separen a su vez los distintos archivos de los casos de uso, de las interfaces para diferentes usuarios, los distintos dominios, etc. A su vez para cada subcarpeta de una capa se tienen un conjunto de archivos .py que contendrán el codigo.
